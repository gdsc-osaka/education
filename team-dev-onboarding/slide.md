# 01. Web アプリ全体像
## 15 分

---

## このセクションのゴール

![bg right:40% fit](../images/data-flow.png)

細かい実装より、

**自分がどこを触っているか分かる**

状態を目指しましょう!

---

## Web アプリは「往復」で動く

![bg right:50% fit](../images/web-app-arch.png)

- ユーザーの**操作**が API としてサーバーに送られる
- サーバーが処理して、DB に**読み書き**する
- 結果が**画面に返ってくる**

> データは一方通行ではなく、**行って帰ってくる**!

---

## Frontend の役割

![bg right:45% fit](../images/frontend-role.png)

ユーザーが**直接触れる**部分です

- 画面の表示
- ボタンのクリック
- フォームの入力

> 例: React, HTML / CSS, JavaScript

---

## Backend の役割

![bg right:45% fit](../images/backend-role.png)

Frontend からの**依頼を処理**する部分です

- データを処理する
- DB / Storage とのやりとり
- ログイン / API の提供

> 例: Node.js, Python, Go

---

## DB（データベース）

![bg right:45% fit](../images/db-role.png)

データを**保存しておく場所**です

- ユーザー情報
- 投稿内容
- 設定

> 消えると困るデータは DB に入れます

---

<!-- _class: lead -->

# 具体例: ツイート投稿ボタンを押すと？

抽象的だったので、Twitter で

「ツイートする」を押した瞬間に
裏で何が起きているかを追ってみましょう

---

![bg fit](../images/twitter-architecture.png)

---

## ① 画面が出来るまで (HTML / CSS / React)

ブラウザでツイート画面を開いた時点でも色々起きています

- **HTML** … 「テキストエリアとボタンがある」骨格を作る
- **CSS** … 「ボタンは青、角丸、文字は白」見た目を当てる
- **React** … 「文字を打ったら state が更新される」動きを乗せる

> ここまでは**自分のブラウザの中だけ**で完結!

---

## ② ボタンを押した瞬間 (React → サーバー)

クリックは**まず React が拾い**、サーバーにリクエストを送ります

```js
// 雰囲気のコード
const onClick = () => {
  fetch('https://api.twitter.com/tweets', {
    method: 'POST',
    headers: { Authorization: `Bearer ${token}` },
    body: JSON.stringify({ text }),
  })
}
```

> DevTools の **Network タブ**に出てくるのがこのリクエスト!

---

## ③ バックエンドサーバーの仕事

サーバー（Node.js / Go など）が受け取ったリクエストを処理します

- **認証** … トークンから「本当に本人か」を確認
- **バリデーション** … 280 文字以内？ 空じゃない？
- **DB に書き込む** … `INSERT INTO tweets ...` を発行

> ここで弾かれると `400` / `401` のエラーがブラウザに返ってきます

---

## ④ DB がツイートを保存する

DB はツイートを**消えないように**しまっておく場所

```sql
INSERT INTO tweets (user_id, text, created_at)
VALUES (123, '今日の昼ごはん', NOW());
```

- 成功すれば**新しい行の ID** が返る
- 失敗すれば**エラー**（容量切れ、接続断 など）

---

## ⑤ 画面に反映される

レスポンスがブラウザに返り、React が画面を更新します

```json
{ "id": 999, "text": "今日の昼ごはん", "createdAt": "..." }
```

- React が**タイムラインの state** に新しいツイートを追加
- **差分だけ** DOM を書き換え → 投稿欄が空に、ツイートが上に出現

> 全部で**コンマ数秒**。これが Web アプリの 1 サイクル!

---

## 自分が触るのはどこ？

| レイヤー       | やること例             |
| ---------- | ----------------- |
| HTML / CSS | ボタンの色、文字の大きさ      |
| React      | クリック時の挙動、画面切り替え   |
| Backend    | 認証、バリデーション、SQL    |
| DB         | テーブル設計、保存形式       |

> 今日触るのは **React のところを少しだけ**!

# 02. 開発フロー体験
## 45 分（中核）

---

## このセクションのゴール

![bg right:45% fit](../images/github-flow.png)

全員で**実際に手を動かしてみましょう**!

- clone → branch → 編集 → commit → push → PR
- 初心者最大の壁「変更 → PR」をまず体験
- 経験者の方は隣でサポートお願いします!

---

## 今日の開発フロー

![bg right:45% fit](../images/dev-flow-overview.png)

この 7 ステップを一通り体験してみましょう!

```text
clone → branch → 編集 →
commit → push → PR → review
```

> 詰まりやすいのは clone と push です! 詰まったらすぐ呼んでください

---

## Step 1. clone する

リポジトリを自分の PC に持ってきます

```bash
git clone https://github.com/gdsc-osaka/react-example.git
cd react-example
```

> 今日はこのリポジトリを使います!

---

## Step 2. ブランチを切る

`main` では作業しません! **最初に**自分のブランチを作りましょう

```bash
git switch -c feature/add-<your-name>
```

> ブランチ名がカブらないよう、`<your-name>` は自分の名前に置き換えましょう

---

## Step 3. 起動する

依存パッケージを入れて、開発サーバーを立ち上げます

```bash
npm install
npm run dev
```

ブラウザで `http://localhost:5173` を開いてみましょう!

---

## ⚠️ スクリーンショット

`npm run dev` 成功時のターミナル出力

（プレースホルダー）

---

## Step 4. 自分の名前を追加

`src/App.jsx` の `participants` 配列に**自分の名前**を追加しましょう

```diff
  const participants = [
    'Alice',
+   'YOUR_NAME',
  ]
```

ブラウザをリロードして、自分の名前が表示されたら成功!

> 全員分の名前が**最終的に main に並ぶ**のがゴールです

---

## Step 5. commit する

変更内容を Git に記録します

```bash
git add .
git commit -m "feat: add <your-name> to participants"
```

> コミットメッセージは「**何をしたか**」を短く書きましょう

---

## Step 6. push する

自分の変更を GitHub に送ります

```bash
git push -u origin feature/add-<your-name>  # 初回
git push                                    # 2 回目以降
```

> 初回は `-u origin <ブランチ名>` が必要です!

---

## ⚠️ スクリーンショット

GitHub の Pull Request 作成画面

（プレースホルダー）

---

## Step 7. PR を作成

GitHub 上で **Pull Request** を作ります

- ブラウザで GitHub を開く
- 「Compare & pull request」ボタンを押す
- タイトルと説明を書いて作成!

# 03. Git 事故対策
## 15 分

---

## このセクションのゴール

![bg right:40% fit](../images/git-branch.png)

完璧に使いこなすより、まずは

**事故りやすいポイントを知る**ところから!

最低限の安全策を押さえておきましょう

---

## ルール 1. main で作業しない

必ず**ブランチを切ってから**作業しましょう!

```bash
git switch -c feature/my-task
```

> ブランチ名は `feature/<やること>` の形が分かりやすいです

---

## ルール 2. 作業前に pull

最新を取ってきてから作業を始めましょう

```bash
git switch main
git pull
git switch -c feature/next-task
```

> 古い状態で作業すると conflict の原因になります...

---

## ルール 3. 困ったら git status

今の状態を確認するクセを付けましょう

```bash
git status
```

- どのファイルを変えたか
- どこまで commit したか
- どのブランチにいるか

---

## ルール 4. conflict は助けを呼ぶ

merge conflict は初心者の鬼門です

- 無理に直そうとしない
- そのまま経験者に画面を見せる
- 自分で扱うのは慣れてからで OK!

---

## 禁止事項

これだけは絶対やめましょう!

```bash
git push --force              # ❌ 履歴を吹き飛ばす
git push --force-with-lease   # ❌ こちらもまず使わない
```

> どうしても必要なときは、**必ず**経験者に確認してください

# 04. DevTools & ターミナル
## 20 分

---

## なぜ大事？

![bg right:40% fit](../images/devtools-illust.png)

AI 活用の**前提となる知識**です

- エラーが読めれば AI に渡せる
- 渡せれば直せる
- 直せれば自走できる!

---

## DevTools の開き方

どのブラウザでも共通です

- **F12** キーを押す
- 右クリック → **「検証」**
- macOS なら **Cmd + Option + I**

---

## 困ったとき → どのタブ？

| 症状                       | 開くタブ      |
| ------------------------ | --------- |
| 画面が**真っ白**               | Console   |
| ボタンを押しても**保存されない**       | Network   |
| **見た目が崩れている**            | Elements  |

> 症状から逆算できるようになりましょう!

---

## Console タブ

JavaScript のエラーや出力を確認できます

- **赤文字**を探す
- まずは 1 行目を読む
- 例: `Uncaught TypeError: ...`

---

## ⚠️ スクリーンショット

Console タブにエラーが表示された画面

（プレースホルダー）

---

## Network タブ

API 通信の様子を確認できます

- リクエストの一覧が見える
- **赤色 / 404 / 500** を探す
- 「サーバーまで届いてる？」が分かる

---

## ⚠️ スクリーンショット

Network タブで失敗リクエストが赤く出ている画面

（プレースホルダー）

---

## Elements タブ

HTML の構造を確認できます

- 今表示されている HTML が見える
- CSS が当たっているか分かる
- クラス名も確認できる

---

## 最重要ポイント

![bg right:40% fit](../images/read-error.png)

**エラー文を読む**

文化を作りましょう!

> 「赤いの出てます」ではなく、「○○というエラーが出ました」と伝えるのがコツ

---

## ターミナルは怖くない

![bg right:40% fit](../images/terminal-friendly.png)

初心者は**ターミナル恐怖症**になりがちです

まず覚えてほしいのはこの 1 つだけ

**壊れても大抵は戻せます!**

---

## 最低限コマンド

| コマンド            | 役割              |
| --------------- | --------------- |
| `pwd`           | 今いる場所           |
| `ls` / `dir`    | ファイル一覧（Win は dir） |
| `cd folder`     | 移動              |
| `cd ..`         | 一つ上に戻る          |
| `Ctrl + C`      | 動いているコマンドを止める   |

> `Ctrl + C` を覚えるだけで安心感が全然違います!

# 05. AI への聞き方
## 15 分

---

## なぜ必須？

2026 年、AI を使いこなせるかどうかは

**生産性に直結する**スキルです

- 最初の取っかかりは AI に聞いて OK!
- ただし最後は**公式ドキュメント / 実行結果 / 経験者**で必ず確認
- 聞き方ひとつで結果が大きく変わります

---

## 悪い例

![bg right:45% fit](../images/ai-bad-question.png)

```text
動きません
```

- AI もエスパーではありません
- 状況が分からないと当てずっぽうの回答に...
- 結局解決しないまま終わってしまう

---

## 良い例

![bg right:40% fit](../images/ai-good-question.png)

```text
React で npm run dev したら
Module not found が出ました

【やりたいこと】
ログイン画面を表示したい

【エラー全文】
xxx

【試したこと】
npm install をやり直した
```

---

## AI 質問テンプレ（コピペ可）

<div class="fit">

```text
【やりたいこと】
例: ログイン画面を表示したい

【起きていること】
例: npm run dev でエラー

【エラー全文】
（そのまま貼る）

【試したこと】
例: npm install をやり直した

【環境】
OS: macOS / Windows
ブラウザ: Chrome
```

</div>

---

## スクショは要約しない

![bg right:40% fit](../images/error-summarize.png)

エラーを**要約してしまう**と伝わりません

```text
なんか赤いエラー出てます
```

→ これだと AI には何も伝わりません...

---

## スクショをそのまま貼る

![bg right:40% fit](../images/screenshot-paste.png)

AI は**画像認識がとても得意**です!

- レイアウト崩れ
- エラー画面 / DevTools
- ターミナル

> 文字起こしより、画像をそのまま投げる方が早いです!

---

## スクショの撮り方

| OS      | 範囲選択              | 全画面             |
| ------- | ----------------- | --------------- |
| Windows | `Win + Shift + S` | `PrintScreen`   |
| macOS   | `Cmd + Shift + 4` | `Cmd + Shift + 3` |

> **範囲選択**の方が圧倒的に便利です!

---

## AI は間違えることがある

![bg right:40% fit](../images/ai-warning.png)

それっぽいコードを返してくれますが、

- 存在しない関数を使う
- 古い API を提案する
- 副作用を説明しない

…ということがあります

---

## 特に AI 任せにしない操作

- ライブラリの追加 / 削除
- DB の操作
- 設定ファイルの大きな変更
- `--force` 系コマンド

> **大きい変更の前には経験者に聞きましょう**! 1 分の確認で半日が救えます

---

## ⚠️ AI に貼ってはいけないもの

事故が起きやすいので、投げる前に必ず確認しましょう!

- **パスワード** / API キー
- **`.env`** の中身
- 個人情報（メール、住所、電話）
- 秘密の URL（社内 / 招待リンク）
- 未公開の本番データ

> 不安なときは**マスクしてから**貼ってください

# 06. チームルール & まとめ

## なぜルールが必要？

![bg right:40% fit](../images/team-rules.png)

学生プロジェクトは**ここで崩壊**しがちです

- 仲が悪いから崩れるわけではありません
- ルールがないから崩れてしまう
- だからこそ最低限の約束を共有しましょう!

---

## 困ったときの判断フロー

1. **エラー文**を読む
2. **スクショ**を撮る
3. **AI** に聞く（テンプレに沿って）
4. **10〜30 分**自分で試す
5. 進まなければ**経験者**に聞く

> 30 分で進まなければ、すぐ声をかけてください!

---

## メンター相談テンプレ

<div class="fit">

```text
【やりたいこと】
【起きていること】
【エラー全文】
【試したこと】
【スクショ】
【今いるブランチ】 ← Git 事故防止
```

</div>

> ブランチ名を添えると、メンターの初動が早くなります!

---

## チーム開発の最低ルール

- **30 分**詰まったらすぐ相談!
- 「動きません」ではなく**エラー全文 + スクショ**を共有
- 勝手に**ライブラリを追加**しない
- **`main` に直 push** しない
- **壊したら隠さない**（次スライドで詳しく）

---

## 壊したら隠さない（最重要）

- 「やっちゃいました」をすぐ報告!
- 早ければ早いほどリカバリが簡単
- 隠すほど被害が広がってしまう

> **隠さないチームは強い**!

---

| 触れない                      | 理由                |

| React 詳細 / TS 厳密 / CSS 設計 | 必要になったとき学ぶ方が定着    |
| DB 理論 / Docker 操作         | まず「参加できる」を優先      |
| アルゴリズム / 設計原則             | 実戦で身につく           |
