# 01. Web アプリ全体像
## 20 分

---

## このセクションのゴール

![bg right:40% fit](../images/data-flow.png)

詳細実装ではなく、

**自分がどこを触っているか分かる**

状態を目指す。

---

## Web アプリの構成

![bg right:45% fit](../images/web-app-arch.png)

データは上から下へ流れる:

- **Browser**（ブラウザ）
- **Frontend**（画面）
- **Backend**（処理）
- **DB**（保存）

---

## Frontend の役割

![bg right:45% fit](../images/frontend-role.png)

ユーザーが**直接触れる**部分。

- 画面の表示
- ボタンのクリック
- フォームの入力

> 例: React, HTML / CSS, JavaScript

---

## Backend の役割

![bg right:45% fit](../images/backend-role.png)

Frontend からの**依頼を処理**する。

- データ処理
- DB / Storage との接続
- ログイン / API 提供

> 例: Node.js, Python, Go

---

## DB（データベース）

![bg right:45% fit](../images/db-role.png)

データを**保存する場所**。

- ユーザー情報
- 投稿内容
- 設定

> 消えると困るものは DB へ。

---

## GitHub の役割

![bg right:45% fit](../images/github-role.png)

チーム開発の中心地。

- コードの共有
- PR レビュー
- 変更履歴の管理

---

## Docker（概念だけ）

![bg right:45% fit](../images/docker-role.png)

**みんなの開発環境を揃える箱**。

- 「うちでは動くのに…」を防ぐ
- 今は使い方を覚えなくて OK
- 「箱で揃える仕組みがある」だけ知る

---

## ここで大事なこと

詳細実装ではなく、

**データがどう流れるか**

を理解しよう。

> 「自分が今どこを触っているか」が言えれば合格！

# 02. 開発フロー体験
## 35 分（メイン）

---

## このセクションのゴール

![bg right:45% fit](../images/github-flow.png)

全員で**実際に手を動かす**。

- clone → 起動 → 修正 → commit → push → PR
- 初心者最大の壁「変更 → PR」を経験する
- 経験者は隣でサポート

---

## Step 1. clone する

リポジトリを自分の PC に持ってくる。

```bash
git clone https://github.com/xxx/yyy.git
cd yyy
```

> URL はチームから配布されます。

---

## Step 2. 起動する

依存パッケージを入れて開発サーバーを立ち上げる。

```bash
npm install
npm run dev
```

ブラウザで `http://localhost:3000` を開く。

---

## ⚠️ スクリーンショット

`npm run dev` 成功時のターミナル出力

（プレースホルダー）

---

## Step 3. 文言を修正

例: ページのタイトルを変えてみよう。

```diff
- <h1>Hello</h1>
+ <h1>Hello World</h1>
```

ブラウザをリロードして変わっているか確認！

---

## Step 4. commit する

変更内容を Git に記録する。

```bash
git add .
git commit -m "fix: change title text"
```

> コミットメッセージは「**何をしたか**」を短く。

---

## Step 5. push する

自分の変更を GitHub に送る。

```bash
git push
```

---

## ⚠️ スクリーンショット

GitHub の Pull Request 作成画面

（プレースホルダー）

---

## Step 6. PR を作成

GitHub 上で **Pull Request** を作る。

- ブラウザで GitHub を開く
- 「Compare & pull request」ボタン
- タイトルと説明を書いて作成

---

## このセクションのゴール（再）

**変更 → PR** を一通り経験できた！

- 最初は緊張するが、誰もが通る道
- 詰まったら即メンターに声かけ
- 「動いた」の感覚を覚えておこう

# 03. Git 事故対策
## 15 分

---

## このセクションのゴール

![bg right:40% fit](../images/git-branch.png)

完璧に使いこなすより、まず

**壊し方を知る**。

事故を踏まないための最低限を学ぶ。

---

## ルール 1. main で作業しない

必ず**ブランチを切ってから**作業。

```bash
git switch -c feature/my-task
```

> `feature/<やること>` の形が分かりやすい。

---

## ルール 2. 作業前に pull

最新を取ってきてから作業を始める。

```bash
git pull
```

> 古い状態で作業すると conflict の原因。

---

## ルール 3. 困ったら git status

今の状態を確認するクセを付ける。

```bash
git status
```

- どのファイルを変えたか
- どこまで commit したか
- どのブランチにいるか

---

## ルール 4. conflict は助けを呼ぶ

merge conflict は初心者の鬼門。

- 無理に直そうとしない
- そのまま経験者に画面を見せる
- 学ぶのは慣れてから

---

## 禁止事項

これだけは絶対やらない。

```bash
git push --force        # ❌ 履歴を吹き飛ばす
git push --force-with-lease  # ❌ こちらもまず使わない
```

> やるなら**必ず**経験者に確認。

# 4-1. ブラウザ開発者ツール
## 10 分

## 開き方

どのブラウザでも共通。

- **F12** キー
- 右クリック → **「検証」**

Chrome DevTools 全体像（タブが見える状態）

（プレースホルダー）

---

## Console タブ

JavaScript のエラーと出力を見る。

- **赤文字**を探す
- まずは 1 行目を読む
- 例: `Uncaught TypeError: ...`

---

## ⚠️ スクリーンショット

Console タブにエラーが表示された画面

（プレースホルダー）

---

## Network タブ

API 通信の様子を見る。

- リクエスト一覧
- **赤色 / 404 / 500** を探す
- 「サーバーまで届いてる？」を確認

---

## ⚠️ スクリーンショット

Network タブで失敗リクエストが赤く出ている画面

（プレースホルダー）

---

## Elements タブ

- 今表示されている HTML

- クラス名は何か

## 最重要ポイント

![bg right:40% fit](../images/read-error.png)

**エラー文を読む**

文化を作ろう。

> 「赤いの出てます」ではなく、「○○というエラーが出ました」。

# 4-2. ターミナル / コンソール
## 10 分

---

## ターミナルは怖くない

![bg right:40% fit](../images/terminal-friendly.png)

初心者は**ターミナル恐怖症**になりやすい。

まず覚えてほしいことは 1 つ:

**壊れても大抵戻せる**

---

## 最低限コマンド: pwd

今いる場所を確認する。

```bash
pwd
```

> 「自分がどこにいるか」を言えるのが第一歩。

---

## 最低限コマンド: ls / dir

その場のファイル一覧を見る。

```bash
ls          # macOS / Linux
dir         # Windows PowerShell
```

---

## 最低限コマンド: cd

別のフォルダに移動する。

```bash
cd folder-name
cd ..        # 一つ上に戻る
```

---

## 最低限コマンド: Ctrl + C

**動いているコマンドを止める**。

```text
Ctrl + C
```

- `npm run dev` を止めるとき
- 反応がなくなったとき
- これが効くと知るだけで安心感が違う

---

## ⚠️ スクリーンショット

ターミナルで `pwd / ls / cd` を実行した出力例

（プレースホルダー）

---

## 伝えたいこと

ターミナルは「黒い画面」ではなく、

**コンピュータとの対話窓口**。

> 壊れても大抵戻せる。怖がらず触ろう。

# 5-1. AI に渡す情報

---

## 悪い例

![bg right:45% fit](../images/ai-bad-question.png)

```text
動きません
```

- AI もエスパーではない
- 状況不明 → 当てずっぽうの回答
- 結局解決しない

---

## 良い例

![bg right:40% fit](../images/ai-good-question.png)

```text
React で npm run dev したら
Module not found が出ました

【やりたいこと】
ログイン画面を表示したい

【エラー全文】
xxx

【試したこと】
npm install をやり直した
```

---

## 良い質問の 3 要素

- **やりたいこと**（目的）
- **起きていること**（エラー全文）
- **試したこと**（やった対応）

> この 3 つを毎回書くだけで劇的に変わる。

# 5-4. AI を盲信しない

---

## AI は平気で嘘をつく

![bg right:40% fit](../images/ai-warning.png)

それっぽいコードを返すが、

- 存在しない関数を使う
- 古い API を提案する
- 副作用を説明しない

ことがある。

---

## 特に危険な操作

これらは AI 任せにしない:

- ライブラリの追加 / 削除
- DB の操作
- 設定ファイルの大きな変更
- `--force` 系コマンド

---

## 基本ルール

**大きい変更の前は経験者に聞く**。

- 「ライブラリ入れていい？」
- 「この config 触っていい？」
- 「DB のテーブル変えていい？」

> 1 分の確認で半日のロスが防げる。

# 06. チームルール
## 10 分

---

## なぜ必要？

![bg right:40% fit](../images/team-rules.png)

学生プロジェクトは**ここで崩壊**しやすい。

- 仲が悪いから崩れるのではない
- ルールがないから崩れる
- 最低限の約束を共有する

---

## ルール 1. 詰まったら 30 分で相談

- 1 人で抱え込まない
- 30 分やって進まなければ声をかける
- 「相談する」は弱さではなくスキル

---

## ルール 2. エラー文を貼る

「動きません」は**禁止ワード**。

- エラー全文を貼る
- スクショを貼る
- 状況を一言添える

---

## ルール 3. 勝手にライブラリを追加しない

- `npm install xxx` の前に相談
- 依存が増えると全員に影響
- セキュリティ的にも要確認

---

## ルール 4. 勝手に main へ push しない

- 必ずブランチで作業
- PR を出してレビューを通す
- 直 push は事故のもと

---

## ルール 5. 壊したら隠さない（最重要）

- 「やっちゃいました」を即報告
- 早ければ早いほどリカバリは簡単
- 隠すと被害が広がる

> **隠さないチームは強い**。
