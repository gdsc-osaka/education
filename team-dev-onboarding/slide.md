# 02. 開発フロー体験
## 45 分（中核）

---

## このセクションのゴール

![bg right:45% fit](../images/github-flow.png)

全員で**実際に手を動かす**。

- clone → branch → 編集 → commit → push → PR
- 初心者最大の壁「変更 → PR」を経験する
- 経験者は隣でサポート

---

## 今日の開発フロー

![bg right:45% fit](../images/dev-flow-overview.png)

7 ステップを一通り体験。

```text
clone → branch → 編集 →
commit → push → PR → review
```

> 詰まりやすいのは clone と push。すぐ呼んで！

---

## Step 1. clone する

リポジトリを自分の PC に持ってくる。

```bash
git clone https://github.com/xxx/yyy.git
cd yyy
```

> URL はチームから配布されます。

---

## Step 2. ブランチを切る

`main` で作業しない。**最初に**自分のブランチを作る。

```bash
git switch -c feature/first-pr
```

> `git status` で今いるブランチが見える。

---

## Step 3. 起動する

依存パッケージを入れて開発サーバーを立ち上げる。

```bash
npm install
npm run dev
```

ブラウザで `http://localhost:3000` を開く。

---

## ⚠️ スクリーンショット

`npm run dev` 成功時のターミナル出力

（プレースホルダー）

---

## Step 4. 文言を修正

例: ページのタイトルを変えてみよう。

```diff
- <h1>Hello</h1>
+ <h1>Hello World</h1>
```

ブラウザをリロードして変わっているか確認！

---

## Step 5. commit する

変更内容を Git に記録する。

```bash
git add .
git commit -m "fix: change title text"
```

> コミットメッセージは「**何をしたか**」を短く。

---

## Step 6. push する

自分の変更を GitHub に送る。

```bash
git push -u origin feature/first-pr  # 初回
git push                             # 2 回目以降
```

> 初回は `-u origin <ブランチ名>` が必要。

---

## ⚠️ スクリーンショット

GitHub の Pull Request 作成画面

（プレースホルダー）

---

## Step 7. PR を作成

GitHub 上で **Pull Request** を作る。

- ブラウザで GitHub を開く
- 「Compare & pull request」ボタン
- タイトルと説明を書いて作成

# 03. Git 事故対策
## 15 分

---

## このセクションのゴール

![bg right:40% fit](../images/git-branch.png)

完璧に使いこなすより、まず

**事故りやすいポイントを知る**。

最低限の安全策を学ぶ。

---

## ルール 1. main で作業しない

必ず**ブランチを切ってから**作業。

```bash
git switch -c feature/my-task
```

> `feature/<やること>` の形が分かりやすい。

---

## ルール 2. 作業前に pull

最新を取ってきてから作業を始める。

```bash
git switch main
git pull
git switch -c feature/next-task
```

> 古い状態で作業すると conflict の原因。

---

## ルール 3. 困ったら git status

今の状態を確認するクセを付ける。

```bash
git status
```

- どのファイルを変えたか
- どこまで commit したか
- どのブランチにいるか

---

## ルール 4. conflict は助けを呼ぶ

merge conflict は初心者の鬼門。

- 無理に直そうとしない
- そのまま経験者に画面を見せる
- 学ぶのは慣れてから

---

## 禁止事項

これだけは絶対やらない。

```bash
git push --force              # ❌ 履歴を吹き飛ばす
git push --force-with-lease   # ❌ こちらもまず使わない
```

> やるなら**必ず**経験者に確認。

# 04. DevTools & ターミナル
## 20 分

---

## なぜ大事？

![bg right:40% fit](../images/devtools-illust.png)

AI 活用の**前提知識**。

- エラーが読めれば AI に渡せる
- 渡せれば直せる
- 直せれば自走できる

---

## DevTools の開き方

どのブラウザでも共通。

- **F12** キー
- 右クリック → **「検証」**
- macOS: **Cmd + Option + I**

---

## 困ったとき → どのタブ？

| 症状                       | 開くタブ      |
| ------------------------ | --------- |
| 画面が**真っ白**               | Console   |
| ボタンを押しても**保存されない**       | Network   |
| **見た目が崩れている**            | Elements  |

> 症状から逆引きできるようになろう。

---

## Console タブ

JavaScript のエラーと出力を見る。

- **赤文字**を探す
- まずは 1 行目を読む
- 例: `Uncaught TypeError: ...`

---

## ⚠️ スクリーンショット

Console タブにエラーが表示された画面

（プレースホルダー）

---

## Network タブ

API 通信の様子を見る。

- リクエスト一覧
- **赤色 / 404 / 500** を探す
- 「サーバーまで届いてる？」を確認

---

## ⚠️ スクリーンショット

Network タブで失敗リクエストが赤く出ている画面

（プレースホルダー）

---

## Elements タブ

HTML の構造を見る。

- 今表示されている HTML
- CSS が当たっているか
- クラス名は何か

---

## 最重要ポイント

![bg right:40% fit](../images/read-error.png)

**エラー文を読む**

文化を作ろう。

> 「赤いの出てます」ではなく、「○○というエラーが出ました」。

---

## ターミナルは怖くない

![bg right:40% fit](../images/terminal-friendly.png)

初心者は**ターミナル恐怖症**になりやすい。

まず覚えてほしいことは 1 つ:

**壊れても大抵戻せる**

---

## 最低限コマンド

| コマンド            | 役割              |
| --------------- | --------------- |
| `pwd`           | 今いる場所           |
| `ls` / `dir`    | ファイル一覧（Win は dir） |
| `cd folder`     | 移動              |
| `cd ..`         | 一つ上に戻る          |
| `Ctrl + C`      | 動いているコマンドを止める   |

> `Ctrl + C` を知るだけで安心感が違う。

# 05. AI への聞き方
## 15 分

---

## なぜ必須？

2026 年において、AI を使いこなせるかは

**生産性に直結**するスキル。

- 最初の取っかかりは AI に聞いて OK
- ただし最後は**公式ドキュメント / 実行結果 / 経験者**で確認
- 聞き方で結果が大きく変わる

---

## 悪い例

![bg right:45% fit](../images/ai-bad-question.png)

```text
動きません
```

- AI もエスパーではない
- 状況不明 → 当てずっぽうの回答
- 結局解決しない

---

## 良い例

![bg right:40% fit](../images/ai-good-question.png)

```text
React で npm run dev したら
Module not found が出ました

【やりたいこと】
ログイン画面を表示したい

【エラー全文】
xxx

【試したこと】
npm install をやり直した
```

---

## AI 質問テンプレ（コピペ可）

<div class="fit">

```text
【やりたいこと】
例: ログイン画面を表示したい

【起きていること】
例: npm run dev でエラー

【エラー全文】
（そのまま貼る）

【試したこと】
例: npm install をやり直した

【環境】
OS: macOS / Windows
ブラウザ: Chrome
```

</div>

---

## スクショは要約しない

![bg right:40% fit](../images/error-summarize.png)

エラーを**要約してしまう**と伝わらない。

```text
なんか赤いエラー出てます
```

→ AI には何も伝わらない。

---

## スクショをそのまま貼る

![bg right:40% fit](../images/screenshot-paste.png)

AI は**画像認識が強い**。

- レイアウト崩れ
- エラー画面 / DevTools
- ターミナル

> 文字起こしより、画像をそのまま投げる方が早い。

---

## スクショの撮り方

| OS      | 範囲選択              | 全画面             |
| ------- | ----------------- | --------------- |
| Windows | `Win + Shift + S` | `PrintScreen`   |
| macOS   | `Cmd + Shift + 4` | `Cmd + Shift + 3` |

> 実務では**範囲選択**が圧倒的に便利。

---

## AI は間違えることがある

![bg right:40% fit](../images/ai-warning.png)

それっぽいコードを返すが、

- 存在しない関数を使う
- 古い API を提案する
- 副作用を説明しない

ことがある。

---

## 特に AI 任せにしない操作

- ライブラリの追加 / 削除
- DB の操作
- 設定ファイルの大きな変更
- `--force` 系コマンド

> **大きい変更の前は経験者に聞く**。1 分の確認で半日が救える。

---

## ⚠️ AI に貼ってはいけないもの

事故が起きやすい。投げる前に確認。

- **パスワード** / API キー
- **`.env`** の中身
- 個人情報（メール、住所、電話）
- 秘密の URL（社内 / 招待リンク）
- 未公開の本番データ

> 不安なら**マスクしてから**貼る。

# 06. チームルール & まとめ
## 10 分

---

## なぜルールが必要？

![bg right:40% fit](../images/team-rules.png)

学生プロジェクトは**ここで崩壊**しやすい。

- 仲が悪いから崩れるのではない
- ルールがないから崩れる
- 最低限の約束を共有する

---

## 困ったときの判断フロー

1. **エラー文**を読む
2. **スクショ**を撮る
3. **AI** に聞く（テンプレに沿って）
4. **10〜30 分**自分で試す
5. 進まなければ**経験者**に聞く

> 30 分で進まなければ即声をかける。

---

## メンター相談テンプレ

<div class="fit">

```text
【やりたいこと】
【起きていること】
【エラー全文】
【試したこと】
【スクショ】
【今いるブランチ】 ← Git 事故防止
```

</div>

> ブランチ名を添えると、メンターの初動が速い。

---

## チーム開発の最低ルール

- **30 分**詰まったら相談
- 「動きません」ではなく**エラー全文 + スクショ**
- 勝手に**ライブラリ追加**しない
- **`main` に直 push** しない
- **壊したら隠さない**（次スライドで詳しく）

---

## 壊したら隠さない（最重要）

- 「やっちゃいました」を即報告
- 早ければ早いほどリカバリは簡単
- 隠すと被害が広がる

> **隠さないチームは強い**。

---

## 今日「教えないこと」

| 触れない                      | 理由                |
| ------------------------- | ----------------- |
| React 詳細 / TS 厳密 / CSS 設計 | 必要になったとき学ぶ方が定着    |
| DB 理論 / Docker 操作         | まず「参加できる」を優先      |
| アルゴリズム / 設計原則             | 実戦で身につく           |
